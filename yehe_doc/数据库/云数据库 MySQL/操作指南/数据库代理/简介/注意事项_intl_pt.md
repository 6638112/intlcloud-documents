
- Quando um endereço de conexão proxy é usado para implementar a separação de leitura/gravação, a consistência de leituras não transacionais não é garantida. Se sua empresa exigir consistência de leitura, você poderá encapsulá-la em transações ou usar a sintaxe de dica.

- Quando um endereço de conexão proxy é usado, `show processlist` irá mesclar os resultados de todos os nós antes de retorná-los.
- Para a instrução `PREPARE`, o proxy do banco de dados primeiro enviará `PREPARE` para todos os nós. Quando chega uma solicitação `EXECUTE` subsequente, ela determina a rota de execução de acordo com o tipo de instrução preparada. Por exemplo, se uma instrução de gravação for preparada, ela enviará a instrução para o banco de dados de origem durante a execução e, se uma instrução de leitura fora das transações for preparada, ela enviará a instrução para uma instância somente leitura.
- Após uma conexão comercial chegar ao proxy de banco de dados, o proxy se conectará à instância de origem e a todas as instâncias somente leitura configuradas. O proxy em si não tem limite no número máximo de conexões, que está sujeito principalmente ao número máximo de conexões da instância do banco de dados de back-end (o menor valor deste parâmetro das instâncias de origem e somente leitura afetará o desempenho do negócio ).
- Após a habilitação do proxy de banco de dados, quando uma instância somente leitura for adicionada ou reiniciada, somente novas solicitações de conexão serão roteadas para ela. Você pode visualizar as métricas de desempenho de cada nó proxy por meio da visão geral ou [monitoramento de desempenho](https://intl.cloud.tencent.com/document/product/236/42055). Se você achar que os números de conexões nos nós estão desbalanceados, pode distribuir as conexões por meio do rebalanceamento.
