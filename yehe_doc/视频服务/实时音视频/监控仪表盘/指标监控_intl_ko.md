Tencent Cloud TRTC [콘솔](https://console.cloud.tencent.com/rav)에 들어가 모니터링이 필요한 애플리케이션 이름을 클릭해 해당 애플리케이션의 상세 페이지에 들어간 후, 좌측의 탐색 바에서 [모니터링 대시보드]를 선택하면, 바로 **통화 품질**을 볼 수 있습니다.
통화 품질 안에 **자세히 보기** 링크를 클릭하면, 바로 이번 통화의 상세 지표 모니터링 곡선을 볼 수 있습니다. 자세한 정보는 다음과 같습니다.

## 사용자 부분
이 부분에서는 동영상으로 멀티미디어 데이터의 흐름을 보여줍니다. 왼쪽에 있는 사용자는 멀티미디어 데이터를 클라우드에 올리고, 클라우드 서버의 전송을 통해 다시 오른쪽에 있는 사용자에게 보냅니다.


## 멀티미디어 비트레이트
- **발신 측 비트 레이트**
초당 발신되는 멀티미디어 데이터가 얼마나 되는지 나타내며, 단위는 kbps입니다. 예를 들어 비트레이트가 800kbps라면 1분 동안 발생되는 데이터양은 대략 800k * 60s / 8 = 6M 바이트입니다.
- **수신 측 비트 레이트**
초당 수신되는 멀티미디어 데이터가 얼마나 되는지 나타내며, 단위는 kbps입니다.



## 비디오 프레임 레이트

- **발신 측 프레임 레이트**
 발신 측 프레임 레이트는 초당 얼마나 많은 화면을 코딩하는지를 나타냅니다. 추천치는 15FPS로, 충분히 원활하게 재생될 뿐 아니라 초당 프레임 수가 많다고 해서 단일 화면의 선명도가 떨어지지도 않습니다.
- **수신 측 프레임 레이트**
수신 측 프레임 레이트는 초당 수신하는 비디오 화면의 프레임 수를 나타냅니다. 수치가 높을수록 원활하게 재생되고, 낮을수록 재생 시 랙이 발생합니다.. 보통 5FPS 보다 낮은 프레임 레이트에서 사용자는 뚜렷한 랙을 느끼게 됩니다.



## 네트워크 패킷 손실률(LOSS)
네트워크 패킷 손실률(LOSS) 수치는 작을수록 좋습니다. 예를 들어 0%의 패킷 손실률은 네트워크가 좋음을 나타내고, 30%의 패킷 손실률은 SDK와 클라우드 서버 간에 10개의 데이터 패키지 중 3개는 버려진다는 것을 의미합니다.



## 네트워크 딜레이(RTT)
네트워크 딜레이(RTT)는 SDK와 서버 간에 한 번 왔다 갔다 하는 데 걸리는 시간을 나타냅니다. 이 수치는 작을수록 좋으며, 일반적으로 50ms 이하의 RTT 가 이상적인 상황이며, 100ms 이상의 RTT는 비교적 큰 통화 지연을 가져옵니다.



## CPU 사용률
'App CPU'는 App의 CPU 사용률을 가리키며, 시스템 CPU는 현재 시스템의 전체 CPU 사용률을 가리킵니다.


## 메모리 점유율

메모리 점유율은 App의 메모리 점유율을 가리킵니다. 현대의 실시간 운영 체제(iOS, Android, Windows)는 복잡하고 다양한 정책 기반의 가상 제어 메커니즘을 갖고 있기 때문에 실제 메모리 점유율은 오르락내리락합니다.
